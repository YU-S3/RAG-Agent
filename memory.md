二、短期记忆处理方案 (Session Context)
目标：确保 Agent 在多轮对话中不“失忆”，能理解指代词（如“那个”、“它”），并记住当前入住期间的临时需求。

1. 技术实现：
   ◦ 滑动窗口机制 (Sliding Window)：保留最近的 N 轮对话（例如最近 10 轮），确保上下文连贯。
   ◦ 会话摘要 (Session Summarization)：当对话轮数超过 Token 限制时，调用 LLM 对早期对话进行摘要压缩（例如：“客人曾询问过泳池位置，并抱怨过空调太热”），将摘要作为上下文输入。
   ◦ 会话 ID (Session ID)：以 RoomNumber + LastName 或 MemberID 生成唯一 Session ID，绑定本次入住的所有交互。
2. 存储架构：
   ◦ 使用 Redis 存储当前会话的对话历史列表（List 结构）。
   ◦ Key 格式：session:{hotel\_id}:{room\_no}:{check\_in\_date}
   ◦ 过期策略：设置 TTL（生存时间），例如退房后 24 小时自动清除原始对话记录，仅保留提取出的偏好数据。
3. 应用场景示例：
   ◦ 客人：“遥控器在哪？”
   ◦ Agent：“在床头柜黑色册子里。”
   ◦ 客人：“没有啊。”（短期记忆让 Agent 知道“没有”指的是“没有遥控器”，而不是没有别的东西）
   ◦ Agent：“那我安排服务员送一个过来。”

   三、长期记忆处理方案 (Guest Profile & Memory)
   目标：实现文档中提到的“枕头软硬偏好”、“忌口提醒”、“生日服务”等跨周期个性化服务。
4. 双层存储架构：
   ◦ 结构化数据 (关系型数据库 MySQL/PostgreSQL)：
   ■ 存储明确的标签和偏好。
   ■ 字段示例：guest\_id, pillow\_preference (硬/软), dietary\_restrictions (过敏源), delivery\_preference(机器人/人工), birthday, language\_preference。
   ■ 来源：前台录入 + Agent 从对话中提取的确切偏好。
   ◦ 非结构化数据 (向量数据库 Milvus/Pinecone)：
   ■ 存储历史对话的语义嵌入（Embedding）。
   ■ 用于检索模糊记忆。例如客人问“我上次住的时候喜欢吃什么？”，向量检索可找到历史对话中关于餐饮的段落。
   ■ 来源：每次对话结束后的关键片段归档。
5. 记忆提取与更新机制 (Memory Extraction)：
   ◦ 异步处理：对话结束后，后台运行一个专门的“记忆提取 Agent"。
   ◦ 提取逻辑：
6. 扫描短期记忆中的对话。
7. 识别潜在偏好实体（如“枕头太软”、“不吃香菜”、“喜欢高楼层”）。
8. 隐私过滤：自动剔除敏感信息（如身份证号、具体银行卡号、政治言论等），仅保留文档规定的“非敏感信息”。
9. 写入长期记忆：更新结构化数据库中的标签，或将重要对话片段向量化存入向量库。
   ◦ 置信度阈值：只有当客人明确表达偏好（如“帮我换个硬枕头”）时才更新档案，避免误判。
10. 记忆检索与注入 (RAG for Profile)：
    ◦ 触发时机：客人再次发起交互（扫码或亮屏）时。
    ◦ 检索流程：
11. 根据客人身份（会员 ID 或 手机号）查询结构化偏好标签。
12. 根据当前查询意图，在向量库中检索相关历史对话（例如客人问“有什么好吃的”，检索历史点餐记录）。
    ◦ Prompt 注入：将检索到的信息放入 System Prompt。
    ■ 示例：“你是一位酒店管家。客人姓沈，偏好硬枕头，对谷物过敏，今天是他的生日。请基于这些信息回答。”
