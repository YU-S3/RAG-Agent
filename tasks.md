# Meta Agent 开发任务清单（可执行）

## 0. 使用说明
- 本清单用于直接落地开发，按优先级从上到下执行
- 每个任务均包含：目标、实现要点、交付物、验收标准、依赖
- 状态建议：`TODO` / `DOING` / `DONE` / `BLOCKED`

---

## 1. 里程碑与优先级

### M1：配置化 MVP（P0）
- 目标：跑通单领域端到端闭环
- 包含：EPIC-0、EPIC-1、EPIC-2
- 里程碑验收：
  - 单请求可完成「规划 → 工具执行 → 结果分析 → 输出」
  - 支持基于 `domain.yaml` 的最小配置驱动
  - 支持低置信度重试，且有最大重试次数

### M2：可插拔与稳态（P1）
- 目标：支持工具热插拔、模型路由与失败降级
- 包含：EPIC-3、EPIC-4、EPIC-6
- 里程碑验收：
  - 新工具接入不改主流程
  - 主模型异常可自动降级
  - 输出格式异常可自动自愈

### M3：生产化（P2）
- 目标：记忆增强、可观测、合规与发布硬化
- 包含：EPIC-5、EPIC-7、EPIC-8
- 里程碑验收：
  - 全链路可观测、可回放、可审计
  - 发布支持灰度与回滚
  - 关键质量指标可持续跟踪

## 2. 详细任务清单

## EPIC-0 项目基线搭建（P0）

### TASK-001 初始化项目目录
- 状态：DONE
- 目标：建立统一目录规范，减少后续重构成本
- 实现要点：
  - 创建目录：`app/`、`domains/`、`tools/`、`prompts/`、`eval/`、`infra/`、`tests/`
  - 约定每个目录职责并固化到 README
- 交付物：
  - 初始目录结构
  - 目录职责说明文档
- 验收标准：
  - 新成员可在 10 分钟内理解项目结构
- 依赖：无

### TASK-002 配置系统初始化
- 状态：DONE
- 目标：支持环境变量 + YAML 的统一配置加载
- 实现要点：
  - 使用 Pydantic Settings 建立配置模型
  - 支持多环境配置：dev/test/prod
  - 对关键字段做类型约束和默认值保护
- 交付物：
  - `settings` 配置模块
  - 示例配置文件
- 验收标准：
  - 缺失必填项时可阻断启动并输出清晰错误
- 依赖：TASK-001

### TASK-003 API 启动入口与健康检查
- 状态：DONE
- 目标：提供可部署、可探活的服务入口
- 实现要点：
  - FastAPI 应用入口
  - `/health`、`/ready` 接口
  - 基础日志与请求 ID
- 交付物：
  - API 服务基础框架
- 验收标准：
  - 本地可启动，探活接口稳定返回 200
- 依赖：TASK-002

- 状态：DONE
- 目标：校验结果质量并决定是否重试
- 实现要点：

### TASK-205 条件边与重试策略
- 状态：DONE

## 3. 指标看板（验收口径）
- 任务完成率：M1 结束 > 70%
- 工具调用成功率：M2 结束 > 85%
- 幻觉率：M2 结束 < 5%

## 4. 执行建议
- 先做 P0 闭环，再扩展 P1/P2，避免前期过度工程化
- 每完成 1 个 EPIC，立即补对应测试与评估样本
- 每周固定回顾指标，及时调整路由、重试、工具策略

## 5. 开工前理解检查（统一入口）
- 业务目标是否明确：本期目标聚焦在哪个里程碑（M1/M2/M3）
- 范围边界是否明确：本次不做哪些能力，避免范围膨胀
- 依赖条件是否满足：模型、存储、向量库、观测组件的接入前置是否可用
- 质量门槛是否明确：本次要达成的指标阈值是否已确认
- 回滚方案是否存在：出现回归时是否可快速撤回变更

---

## 6. TASK 执行模板（统一过程）

### 6.1 执行步骤
- 第一步：确认任务输入（需求、依赖、约束）
- 第二步：实现最小可运行版本（先可用，再完善）
- 第三步：补齐测试（单测/集成/回归）
- 第四步：对照验收标准逐项自检
- 第五步：更新状态与沉淀文档

### 6.2 交付物模板
- 代码变更：模块名、核心函数、接口定义
- 配置变更：新增字段、默认值、兼容策略
- 测试变更：新增用例、覆盖路径、失败样例
- 文档变更：使用说明、边界行为、已知限制

### 6.3 完成定义（DoD）
- 功能符合任务目标与边界
- 验收标准全部通过且可复现
- 关键路径有自动化测试覆盖
- 无阻断级缺陷与明显安全风险
- 文档更新完成，状态从 `DOING` 更新为 `DONE`

## 7. 里程碑出入口检查（Entry / Exit）

### 7.1 M1 出入口检查
- Entry：
  - 目录结构、配置系统、API 入口已就绪
  - 至少 1 个领域配置与 2 个工具样例已准备
- Exit：
  - 单领域闭环稳定运行
  - 重试逻辑生效且不超过上限
  - 输出满足 schema 约束

### 7.2 M2 出入口检查
- Entry：
  - M1 通过并具备基础回归测试
  - 工具协议抽象方案已确定
- Exit：
  - 工具热插拔可用
  - 路由降级与预算控制可用
  - 自愈链路可收敛

### 7.3 M3 出入口检查
- Entry：
  - M2 指标稳定并有持续监控
  - 安全与审计方案已落地基础能力
- Exit：
  - 全链路可观测、可审计、可回滚
  - 发布策略可执行并经演练
  - 关键指标达到目标阈值

## 8. 测试与检查矩阵（按能力维度）
- 配置加载：正常配置、缺失必填、非法类型、版本冲突
- 流程编排：正常路径、低置信度重试、重试上限、异常终止
- 工具调用：成功、超时、参数非法、权限不足、幂等重试
- 路由治理：主模型失败降级、预算超限降级、缓存命中与失效
- 记忆检索：短期上下文连续性、长期检索相关性、压缩后可读性
- 安全合规：输入过滤、越权拦截、审计落库完整性

---

## 9. 风险清单与触发信号
- 风险：上下文膨胀导致延迟上升
  - 信号：P95 持续走高、单请求 token 异常增加
  - 动作：启用压缩检索、增加摘要节点、限制历史窗口
- 风险：工具调用不稳定
  - 信号：工具成功率下降、超时率提升
  - 动作：重试退避、幂等键、熔断与降级
- 风险：模型成本失控
  - 信号：单日成本偏离基线、缓存命中率下降
  - 动作：路由降级、预算阈值收紧、提升缓存覆盖
- 风险：质量评估失真
  - 信号：线上反馈恶化但离线评估正常
  - 动作：扩充数据集、增加对抗样本、提高回归频率

---
