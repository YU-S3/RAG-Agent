# 项目容器化与跨机器复用可行性评估计划

## Summary

* 目标：判断当前项目是否能实现“环境打包进 Docker，代码上传 GitHub，另一台机器仅需拉镜像+拉代码+配置自己的 API Key 即可使用”。

* 结论（先行）：能做到，但需要补齐容器化与配置解耦的最小改造；当前仓库尚未具备开箱即用的一键跨机复用能力。

* 交付边界：本阶段仅输出评估与实施计划，不进行代码改动。

## Current State Analysis

* 仓库现状：未发现 `Dockerfile` 与 `docker-compose`，当前默认是本机 Python 环境运行。

* 运行入口明确：后端通过 `uvicorn app.main:app` 启动，应用为 FastAPI，适合容器化。

* 配置机制可用：`app/core/settings.py` 已用 `BaseSettings + .env`，支持环境变量覆盖，具备容器注入配置基础。

* 模型调用链支持多后端：`app/services/llm_router.py` 优先远程 OpenAI 兼容接口，失败再降级本地模型；`domains/default/domain.yaml` 仍默认 `ollama + 127.0.0.1:11434`。

* 存储行为：RAG/记忆/审计等落在项目内 `eval/*` 文件（并可选 Qdrant）；容器中需挂载持久卷避免数据丢失。

* 关键阻塞：

  * 缺少容器构建与运行编排文件。

  * 默认配置含本机回环地址与示例密钥，不适合直接跨机复用。

  * 若追求“单容器含 Ollama + 模型”，需额外处理多进程、镜像体积、模型拉取与算力要求。

## Proposed Changes

* `Dockerfile`（新增）

  * 做什么：构建 Python 3.10 运行镜像，安装 `requirements.txt`，复制项目代码，默认启动 `uvicorn app.main:app`。

  * 为什么：提供可分发、可复用的统一运行环境。

  * 怎么做：设置工作目录为项目根；暴露 `8000`；启动参数绑定 `0.0.0.0`。

* `docker-compose.yml`（新增）

  * 做什么：定义应用服务、环境变量注入、端口映射、卷挂载（至少 `eval/`、可选 `RAG_test001_api_key.txt`）。

  * 为什么：让“拉镜像 + 拉代码 + 填配置”流程标准化，降低跨机器启动复杂度。

  * 怎么做：支持两种模式切换：

    * 模式A（推荐）：应用容器 + 远程模型 API（仅注入 `REMOTE_LLM_*`）。

    * 模式B（可选）：同机 Ollama（同 Compose 增加 Ollama 服务，或单容器多进程方案）。

* `.env.example`（更新）

  * 做什么：移除真实密钥示例，改为占位符；补充容器场景关键项（如 `APP_HOST=0.0.0.0`、`REMOTE_LLM_*`、`RAG_STORE_PROVIDER`）。

  * 为什么：防止敏感信息泄露并确保跨机配置清晰。

  * 怎么做：保留字段完整性，强化注释说明“必须由用户自行填写”。

* `domains/default/domain.yaml`（更新）

  * 做什么：将默认模型策略调整为更适配容器的模式（如优先远程模型，或将 `base_url` 改为可容器网络访问地址）。

  * 为什么：避免容器内默认访问 `127.0.0.1:11434` 造成不可用。

  * 怎么做：通过 `provider/base_url/model` 的合理默认值 + 文档说明，避免误配。

* `README.md`（更新）

  * 做什么：新增“跨机复用流程”章节（构建/发布镜像、GitHub 拉代码、环境变量配置、启动与验收）。

  * 为什么：把“可行”变成团队可执行 SOP。

  * 怎么做：提供最小步骤与故障排查（端口、模型连通性、卷权限、API Key）。

## Assumptions & Decisions

* 假设用户目标是“低成本跨机器可复用”，不是强制“单容器内必须包含大模型权重”。

* 决策1：主路径采用“应用容器化 + 远程模型 API”作为首选，成功率高、镜像轻、运维成本低。

* 决策2：保留“同机 Ollama”兼容路径，但不作为默认；若坚持单容器全内置，可实现但代价高（镜像体积、启动耗时、硬件依赖更强）。

* 决策3：数据目录采用卷持久化（`eval/*` 至少持久化），保证会话/RAG/审计数据在容器重建后保留。

* 决策4：密钥与环境差异全部外置到 `.env` 或运行时注入，不写入镜像与仓库默认配置。

## Verification Steps

* 容器构建验证：`docker build` 成功，镜像可启动并监听 `8000`。

* API 健康验证：`GET /health`、`GET /ready` 返回正常。

* 生成链路验证：`POST /v1/generate` 在目标机器可得到有效输出（远程或本地模型二选一）。

* RAG 验证：上传文档并查询，确认 `eval/rag` 或 Qdrant 写入正常。

* 跨机复现验证：在全新机器执行“拉镜像 + 拉代码 + 填 `.env` + 启动”流程，结果一致。

* 安全验证：默认样例不包含真实密钥，运行时密钥仅来自用户本地注入。
